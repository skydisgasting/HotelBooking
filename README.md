# Бизнес-функция: Бронирование отеля в сервисе туроператора

## 1. Конкурентный анализ

Для анализа были рассмотрены сервисы бронирования путешествий:

* Booking.com
* Ostrovok.ru
* OneTwoTrip
* Яндекс Путешествия

Общие возможности конкурентов:

1. Поиск отелей по направлению и датам.
2. Фильтрация результатов по цене, рейтингу и услугам.
3. Просмотр подробной информации об отеле.
4. Онлайн-бронирование номера.
5. Подтверждение бронирования по электронной почте.
6. Отображение информации о правилах отмены.

На основании анализа определена бизнес-функция «Бронирование отеля».

# 2. Глоссарий

| Термин                     | Определение                                       |
| -------------------------- | ------------------------------------------------- |
| Пользователь               | Клиент сервиса туроператора                       |
| Отель                      | Объект размещения, доступный для бронирования     |
| Номер                      | Размещение определенной категории в отеле         |

| Заказ                      | Совокупность данных о бронировании                |
| Подтверждение бронирования | Уведомление об успешном создании заказа           |
| Поставщик                  | Внешняя система, предоставляющая данные об отелях |

---

# 3. Бизнес-цель

Предоставить пользователям возможность самостоятельно бронировать отели через онлайн-сервис туроператора без обращения к менеджеру.

Ожидаемый результат:

* увеличение количества онлайн-заказов;
* сокращение нагрузки на сотрудников поддержки;
* повышение конверсии посетителей сайта в клиентов.

---

# 4. Описание бизнес-процесса

## Основной сценарий

1. Пользователь открывает страницу поиска отелей.
2. Указывает:

   * направление;
   * даты проживания;
   * количество гостей.
3. Система отображает доступные варианты.
4. Пользователь выбирает отель.
5. Пользователь выбирает номер.
6. Система отображает стоимость и условия бронирования.
7. Пользователь вводит контактные данные.
8. Пользователь подтверждает бронирование.
9. Система создает заказ.
10. Система отправляет подтверждение пользователю.

## Альтернативный сценарий

### Нет доступных номеров

1. Пользователь выполняет поиск.
2. Система не находит доступных вариантов.
3. Пользователю отображается сообщение:
   «На выбранные даты свободных номеров нет».

### Ошибка поставщика

1. Пользователь подтверждает бронирование.
2. Система не получает подтверждение от поставщика.
3. Пользователю отображается сообщение об ошибке.
4. Заказ не создается.

# 5. User Story

**US-001**

Как путешественник,

Я хочу забронировать номер в отеле через сайт туроператора,

Чтобы заранее обеспечить себе проживание в выбранном месте отдыха.

---

# 6. Функциональные требования

## FR-001 Поиск отелей

Система должна предоставлять возможность поиска отелей по следующим параметрам:

* направление;
* дата заезда;
* дата выезда;
* количество гостей.

## FR-002 Просмотр результатов поиска

Система должна отображать список доступных отелей с указанием:

* названия;
* стоимости;
* рейтинга;
* основных услуг.

## FR-003 Бронирование номера

Система должна позволять пользователю выбрать номер и создать бронирование.

## FR-004 Подтверждение бронирования

После успешного создания заказа система должна отправить подтверждение на электронную почту пользователя.

---

# 7. Нефункциональные требования

## NFR-001 Производительность

Результаты поиска должны отображаться не более чем за 5 секунд.

## NFR-002 Доступность

Доступность сервиса должна составлять не менее 99,5% в месяц.

## NFR-003 Безопасность

Передача персональных данных должна осуществляться по HTTPS.

## NFR-004 Удобство использования

Процесс бронирования должен содержать не более 5 шагов после выбора отеля.

---

# 8. Ограничения

1. Бронирование доступно только для существующих предложений поставщиков.
2. Пользователь обязан указать корректный адрес электронной почты.
3. Минимальная длительность проживания определяется правилами отеля.
4. Стоимость может измениться до момента подтверждения бронирования поставщиком.

---

# 9. Критерии готовности (Definition of Ready)

Задача считается готовой к разработке, если:

* бизнес-цель согласована;
* описан основной бизнес-процесс;
* сформулирована User Story;
* определены функциональные требования;

* описаны ограничения;

---

# 10. Критерии приемки

## AC-001

Дано пользователь открыл страницу поиска

Когда заполнены обязательные параметры поиска

Тогда система отображает список доступных отелей.

## AC-002

Дано пользователь выбрал номер

Когда нажата кнопка «Забронировать»

Тогда создается заказ и отображается номер бронирования.

## AC-003

Дано заказ успешно создан

Когда бронирование подтверждено

Тогда пользователю отправляется электронное письмо с подтверждением.

---

# 11. Критерии завершенности (Definition of Done)

Функция считается завершенной, если:

1. Реализованы все функциональные требования.
2. Выполнены критерии приемки.
3. Проведено функциональное тестирование.
4. Критические дефекты отсутствуют.
5. Подготовлена пользовательская документация.
6. Проведен code review.
7. Изменения развернуты на тестовом окружении.
8. Заказчик подтвердил соответствие требованиям.
